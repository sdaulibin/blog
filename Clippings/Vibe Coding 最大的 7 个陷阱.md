很多人以为 AI 编程让人人都变成了工程师。

其实不是。

它只是让“没想清楚”这件事，暴露得更快。

我拆了最近一年很多团队用 Cursor、Copilot、Claude Code 做产品的真实流程，也看了 GitHub Copilot 那个“简单任务快 55.8%”的实验。

看完这篇，你会明白一件事：

> AI 写代码越快，越考验人类的判断力。

## 速度

Vibe Coding 的爽感很容易骗人。

一句话生成页面。

十分钟跑出 Demo。

半天做出一个能点的 MVP。

看起来像开了外挂。

但软件不是 PPT。

GitHub Copilot 早期研究里，开发者完成一个简单 HTTP Server 任务确实快了 55.8%。

问题是，实验测的是“完成速度”，不是“半年后的维护成本”。

一个登录页生成快 10 倍，不代表权限系统、数据流、异常处理、测试覆盖也强 10 倍。

所以第一条结论很简单：

> AI 可以加速产出，但不能自动提高判断质量。

## 腐化

过去一个人写烂代码，烂得有限。

现在一个 AI Agent 可以在 20 分钟里生成 8 个文件、3 套状态管理、2 个重复 API 封装，再顺手绕开原来的架构。

这才是 Vibe Coding 的真实风险。

不是它写不出来。

而是它太能写了。

你没定义边界，它就补一层。

你没说明数据模型，它就猜一个。

你没告诉它已有封装，它就重新造一个。

一个 Bug 修出 3 个 Bug，通常不是模型太笨，而是系统已经没有清晰约束了。

所以不要问：

AI 能不能写代码？

要问：

它写进去的每一行，未来谁来理解、谁来维护、谁来负责？

## 责任

很多“人类无用论”忽略了一件事：

> AI 不承担后果。

产品卖不出去，AI 不扣工资。

用户流失，AI 不开复盘会。

线上事故，AI 不半夜接电话。

LLM 可以模仿“我理解用户”，但它没有真正被用户骂过，没有见过客服群里的崩溃，也没有经历过一次需求方向错了导致 3 个月白干。

产品判断来自真实世界的摩擦。

用户访谈、现场观察、数据回看、商业约束，这些东西不是让模型“生成一个用户画像”就能替代的。

如果你的用户画像只有姓名、年龄、职业、痛点，但没有真实访谈来源，那它只是格式正确的幻想。

真正的产品能力，是把真实的人、真实的场景、真实的痛苦，压缩成一个可以交付的功能边界。

## 顺序

Vibe Coding 最大的误用，是把它当成“直接开干”的按钮。

正确顺序应该反过来：

> 先定义问题，再让 AI 参与方案。

比如你要做一个 NAS 文件清理功能。

烂提示词是：

帮我做一个清理缓存的功能。

好提示词是：

用户每天产生 10GB 视频素材。 NAS 空间达到 90% 时，他需要在导入新项目之前，一键排除 .tmp 和 .cache 文件夹。 请先列出你需要澄清的问题， 再评估这个功能对数据结构、权限、误删风险和回滚机制的影响。

前者会得到一个看起来能用的按钮。

后者才可能得到一个能进产品的方案。

你给 AI 的不应该是愿望。

而应该是约束。

## 流程

很多团队一用 AI，就开始反流程。

不写需求。

不做原型。

不评审。

不写测试。

理由是：

> 现在很快，先做出来看看。

这句话很危险。

传统研发里，产品、设计、工程、测试不是低效摆设，而是分布在不同阶段的风险控制器。

AI 时代这些环节不是消失了，而是更需要前置。

因为生成速度越快，错误扩散越快。

一个没验证的需求，过去可能浪费 2 周开发。

现在可能 2 天就变成一个完整但没人用的系统。

所以 Vibe Coding 真正该省下来的时间，不是拿去塞更多功能。

而是拿去做 4 件事：

澄清需求 修补架构 补测试 写文档

## 模型

不要迷信单一模型。

不同模型的强项不一样。

有的适合读旧代码。

有的适合写测试。

有的适合重构。

有的适合解释复杂上下文。

更重要的是，模型版本会变。

今天表现稳定，明天可能因为厂商调了推理策略、压缩了输出、改了系统提示词，突然开始偷懒。

只看榜单也不够。

跑分测的是标准题，你的项目是混合题。

真实工作里，你需要记录：

哪个模型适合需求澄清 哪个模型适合代码审查 哪个模型容易编造结论 哪个模型喜欢用空话糊弄提交说明

把模型当成“实习生团队”管理。

不要把它当成神。

## 规则

提示词会失效。

规则不会。

如果你不想让 Agent 到处复制粘贴，就加相似度检测。

如果你不想让它逃避类型约束，就禁掉 any、as unknown as、随手 eslint disable。

如果你不想让它乱写注释，就规定注释只能解释“为什么”，不能写历史小作文。

如果你不想它修 Bug 修到别处去，就要求：

先写失败测试 再修代码 最后用测试证明 Bug 消失

对 Agent 来说，Linter、测试、Hook、审查文档，就是最便宜的管理制度。

不要指望每次都靠一句：

> 请认真一点。

## 清单

这套 Vibe Coding 工作流，建议直接保存：

1\. 先写问题，不先写功能 2. 先问用户，不先问模型 3. 先定边界，不先堆页面 4. 先做低保真原型，再写正式代码 5. 每个功能只解决一个明确场景 6. 每次改动必须有测试证明 7. 所有关键决策写进文档 8. 至少用 2 个模型交叉审查 9. 能用规则拦截的问题，不靠提示词提醒 10. AI 负责加速，人负责判断和后果

## 敏捷

有人会说：

> 你这套太重了，不敏捷。

但敏捷从来不是“不想清楚就开干”。

敏捷的本质是双轨：

Discovery 决定做什么 Delivery 决定怎么做

产品团队可以一边验证下一批需求，工程团队一边交付上一批已验证需求。

但在单个功能上，顺序不能乱。

问题空间没回答，就进入方案空间，本质上是在用代码填认知漏洞。

AI 只会让这个漏洞更快变大。

## 红利

普通人用 Vibe Coding，喜欢让它加功能。

高手用 Vibe Coding，优先让它还债。

比如：

让它读完代码后画数据流 让它找重复实现 让它补边界测试 让它把旧逻辑整理成文档 让它指出需求里没定义的异常情况 让它审查“这个功能会影响哪些模块”

这才是 AI 编程的真正红利。

不是让你少思考。

而是把你从机械劳动里释放出来，去做更高质量的思考。
