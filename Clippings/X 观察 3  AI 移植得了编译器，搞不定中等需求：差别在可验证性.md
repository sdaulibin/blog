> X 观察系列第三篇。这一周多，我陆续读到八篇东西：三篇是檄文（Warp 的工厂备忘录、一篇讲 loop engineering 的科普、Karpathy 对 Claude Tag 的评价），五篇是田野报告（一篇对 Claude Tag 的经济批评、HuggingFace 的百 agent 协作实验、Flow 把 OCaml 移植成 Rust、字节洪定坤的 AI Coding 演讲、LiteLLM 把网关迁到 Rust）。它们看起来各说各话，其实是同一个论断在不同海拔、不同任务上的展开。

先抛一个这一周最让我停下来的反差。

同一周，Flow 团队宣布把整个 OCaml 类型检查器 AI 辅助移植成了 Rust，已经上线，自测快 2 倍；LiteLLM 宣布把网关迁到 Rust，自测吞吐 15 倍。两个项目都把一个几万行的成熟系统交给 AI 做了大改造，而且成了。与此同时，字节的洪定坤在自家大会上拿出一组实测：一个豆包的中等难度需求，三个主流模型配三个主流框架，同一个 Prompt 跑了 900 次，功能正确率都到 80% 以上，但只要看可维护性、是否复用了已有组件、是否符合团队规范、有没有破坏历史功能这些维度，得分普遍塌到不及格，而且塌得「非常强的随机性」。

一边是 AI 重写整个编译器都稳，一边是 AI 连一个中等需求都做不到可靠交付。这不是模型能力的问题，三个项目用的是同一代模型。差别在任务本身。这篇要论证的就是这个差别叫什么，以及它如何决定了 AI 工程的全部活计。

## 一、工程离开了模型，落在 C / B / V

那三篇檄文，说的是同一件事，只是站在不同海拔。Warp 站在组织海拔，说工程师别再以为自己在改产品，去建那个生产产品的工厂；loop engineering 站在单个工程师海拔，说重心一直在「从模型本身漂走」，harness 比模型更重要；Karpathy 站在产品海拔，说 Claude Tag 是第三范式，而模型（Opus 4.8）没有变，所有工作都是底层工程。三个海拔，一个向量：**工程的重心离开了模型，迁移到模型之外。**

问题是，大家都知道工程离开了模型，却没人说清它去了哪里。过去两年，模型之外那片空间被反复改名：prompt engineering、context engineering、harness engineering、loop engineering，到产品层叫第三范式，到组织层叫工厂。我在《智能体软件工程》系列里用的 CBV 框架，正好是这片空间的坐标。模型决定「做什么」，模型之外的工程分三根轴：Contract（契约，约束意图与规格）、Boundary（边界，约束它能碰到什么）、Verification（验证，判断产出对不对）。每个流行词其实各自重新发现了其中一根轴：loop engineering 的「能说不的东西」是 V，Claude Tag 的底层工程主要是 B，Warp 的工厂是整条 C 到 B 到 V 被画成了组织架构图。

这就是上半句。这一篇要做的，是在这三根轴里把承重的那一根单独拎出来。

## 二、还有一层：价值也离开了模型

在往下走之前，先补一根第三篇之后才补上的平行线，因为它和后面的论证落在同一个地方。

离开模型的不只是工程，还有价值捕获。那篇对 Claude Tag 的经济批评指出两件事。一是锁定，根在默会知识。Claude Tag 共享同一个实例、不用你教就吸收上下文、自己行动、异步行动，这四点合起来把它从工具变成了同事。而持续的频道记忆，让 Claude 一边跟着频道走，一边把组织里那些没写进文档的 know-how（上次为什么这么决定、这个词在我们这儿什么意思、哪件事找谁）沉淀进它自己的记忆。这是一个 Polanyi 式的翻转：我们知道的比能说出来的多，组织真正靠的是这层说不出来的默会知识，而它过去恰恰是 AI 的弱点；现在它成了护城河。二是定价，AI 按它产出的每个 token 计费，又能无人触发地异步做几乎无限的活，于是企业的成本问责从一个有负责人的反馈回路（按人头预算，谁花谁解释），退化成一个被多人共享冲淡的、只能「打满就全关」的粗糙开关。

把这层和工程层叠起来，会看到一件事：它们落在同一个地方。模型之外那层组织专属的身份与记忆，同时是最难的工程（B 轴）、最大的攻击面（安全）、护城河（锁定）、以及价值捕获的入口（定价）。这不是巧合。工程走到哪，价值就跟着工程走，风险也跟着工程走。

这层平行线先放在这里。接下来要回到工程，因为这一篇真正要挖的，是那三根轴里最脆的一根。

## 三、三根轴里，承重的是 V

为什么是 V？

因为 C 和 B 都是你可以一次性写下来的东西。Contract 是规格、是架构约束、是团队规范，你能把它写成文档、写成类型、写成 lint 规则。Boundary 是权限、是白名单、是沙箱，你能把它配置死。它们都是**声明式**的：写好了，就摆在那里起作用。

V 不一样。Verification 要回答的是「这个产出对不对」，而这个判断的难易，不取决于你写了多少验证代码，取决于这个任务**有没有一个能判定对错的东西**。对有些任务，这个东西现成、廉价、确定（一个跑通的测试、一个和参照逐字节的比对）；对另一些任务，它根本不存在（一个设计好不好、一段交互顺不顺、一段文案有没有说服力）。

所以我要下的核心判断是：**可验证性是任务自带的属性，不是一道你总能工程进去的工序。** 你能把 Contract 写得更细，能把 Boundary 围得更死，但你不能凭空给一个本质上主观的任务造出一个客观的 oracle。一个任务能不能交给 AI、能交到多深，最终由它自带的可验证性封顶。

这句话听起来抽象。这一周这五份田野报告，正好把它从抽象变成了实测，因为它们恰好分布在可验证性这根谱的不同位置上。下面顺着这根谱，从最强锚点端走到最弱锚点端。

为了后面好对照，先把我一直用的 anchor 分类法摆出来：判定一个产出对错所依据的锚点，从强到弱大致分五档，A 类是可执行、确定性的（测试、编译、和参照比对），往下经过可静态检查的、有文档可依的、要靠社区逆向的，到 E 类是纯主观的（好不好、优雅不优雅）。一个任务里，不同维度往往挂在不同档的锚点上。AI 在强锚点的维度上稳，在弱锚点的维度上塌，塌的方式是高方差。这就是那根谱。

## 四、强锚点端：移植，以及为什么 AI 敢动整个系统

谱的最强端是移植。Flow 和 LiteLLM 是这一周的两个样本，它们一起把「为什么 AI 敢做大改造」讲透了。

移植是软件里唯一一种「正确行为有完整可执行规格」的任务，那个规格就是原来的程序。任何「这里该怎样」的问题，都有 ground truth：原程序怎么做的？验收标准也现成：原程序必须过的测试，新程序也必须过。这是最强的一种 V，它判的不是「写得好不好」（E 类），是「跟参照一不一致」（A 类，而且是一个完整参照撑着的强 A 类）。AI 在这里好用，不是因为它聪明，是因为任务把 oracle 白送了。

有意思的是，Flow 和 LiteLLM 选了**相反**的迁移策略，而深层是同一招。

Flow 选 line-by-line 移植，不重写。理由是保住 oracle：逐行翻译，行为不变，「和参照一致」这个廉价的检查才有效；一旦重写，行为会发散，你就分不清哪是 bug、哪是有意改动。换句话说，line-by-line 这条纪律，是用来让验证保持廉价的。翻成 CBV：**Contract 是为了保 V。** 他们还配了两个手法：让 AI 把对应的 OCaml 原码作为注释贴在 Rust 旁边，「不老实的移植无处可藏」，把「有没有自作主张」从一个主观判断变成一个 diff；以及对 AI 下 no-unsafe 硬规则，少数 unsafe 块由人手写，把危险能力从 AI 手里围起来（这是干净的 Boundary）。

LiteLLM 不逐行移植，是按职责切层。它的核心设计叫 clean split：建一个纯 Rust core，只做数据变换（把请求转成 provider 请求、把响应转回来、转 streaming chunk、数 token、归一化错误），而这个 core「从不打开 socket、不读 secret、不写数据库」，I/O 全由 host 进程做。它给这个原则起了个名字，叫「describe, don't execute」。这同样是 Boundary 轴的教科书实现，而且和 Claude Tag 那套是同构的：Claude Tag 是模型决定做什么、Agent Proxy 在边界注入凭证、模型碰不到 vault；这里是 Rust core 决定请求长什么样、却碰不到 I/O 与 secret。两者是同一个模式：把决策和副作用物理隔开。把纯变换核拎出来，意味着 Rust 能一段段塞进生产，而不碰服务器和 I/O 安全面。

LiteLLM 保 V 的手法是 parity check：每个 provider 上线前必须逐字节匹配旧路径的输出，flag 一关就走回老路。Flow 的 oracle 是原程序的源码（静态对照），LiteLLM 的 oracle 是原程序的运行时输出（动态对照）。翻成我那个验证层级（动态线上抓取 > 静态源码分析 > 文档 > 社区逆向 > LLM 推断），两个项目正好取了偏静态和偏动态的两档，但都把「和参照一致」这个强锚点焊死了。

更值得记的是 LiteLLM 的迁移**顺序**，因为它本质是按锚点强度排程的。它先迁 OCR（输出小而确定，无 streaming、schema 小，逐字节最好比，强锚点），再迁 /v1/messages（引入 streaming，时序让逐字节比对变难，锚点变弱），最后迁 /chat/completions（tools、function calling、multimodal、整个 optional-param 矩阵，状态空间最大、锚点最弱）。它没用「锚点强度」这个词，但它的风险排序完全按可验证性来：最可验证的先上，最不可验证的压到最后、且只在前面验证打牢之后才碰。这就是「围锚点」的正面操作。

这两篇还顺手纠正了 loop engineering 那句「harness 比 model 更重要」。Flow 写得很具体：parser 阶段用 Sonnet 4.5，一次只能移植几个函数，得人工逐批喂；checker 阶段换成 Opus 4.5/4.6，「能自治地跑大得多的负载、跑很久」，于是上了 agent team，一个月搞定整个类型检查器。模型一变强，他们就把委托量调上去。所以准确的关系不是哪个更重要，是：**委托量随模型能力上升，但被验证能力封顶。** 模型变强就多放手，同时把那套 C 加 B 加 V 的纪律摁住不变。

Flow 还划了 AI 的边界，很诚实：性能优化阶段用了 Ralph-loop（目标清晰、改进无穷），AI 能持续找出 1%、2% 的、人会当噪声丢掉但累积可观的微优化；但好几次 AI 找了几天什么也没找到，最后是人靠洞察找到大机会（work-stealing 调度器）或大缺陷（被隐式重复的计算）。Ralph-loop 收割的是局部梯度，架构级的跳变得人来给新结构。这条分工后面还会用到。

给这两篇该有的信用：它们都是厂商讲自己的成功故事（LiteLLM 的 SEO 味尤其重，标题只喊 15 倍，但 FAQ 里自己承认那个 7.5ms 的 Python 基线是没调优的，而且网关开销通常只占模型总延迟一小部分，所以倍数有水分、感知场景有限）。但它们也是这批材料里最经得起查的：LiteLLM 把 benchmark 脚本、mock、CSV 全 check-in 进 GitHub，明说只测转发路径；Flow 老老实实交代了哪些是 AI 干的、哪些是人补的。这个诚实从哪来，后面第七节会回答。

## 五、有数字，但塌在噪声和漏洞上：百 agent 实验

谱往中间走一档：有一个数字 oracle，但它噪声大、还能被钻空子。HuggingFace 那场百 agent 协作实验是这一档的样本。一百多个 agent 开放协作一周，把 Gemma 4 在 vLLM 上的推理速度提了 5 倍。它整场戏的看点，全在验证的完整性上，正好对应我第三篇里那个抽象论证的三种失败模式，而且是活的。

一是弱锚点造成的漏洞。PPL 是 teacher-forced 的，对解码发散是盲的，于是有 agent 用干净的 PPL 把 TPS 刷上去。翻成 anchor 分类法：TPS 和 PPL 都是 A 类锚点，但 PPL 是一个**弱** A 类，它能跑，却不捕捉真正的目标。漏洞就长在这个弱锚点和真实目标之间的缝里。这说明「可验证」内部还有一层等级：不是所有能执行的检查都同等地抓得住 done。

二是 Goodhart，被实时看到。把 lm\_head 剪到一个用公开 eval 建的 keep-set，导致私有子集退化。优化代理指标，目标退化，教科书级，只不过是活的。

三是噪声，这条最深也最被低估。有 agent 把第一名跑了四遍，发现单次 σ≈1.16 TPS；另一个在 358 次运行、66 个桶上确认，前沿差距小于约 4 TPS 的都是平局。也就是说，这场竞赛的大部分「进步」是测量噪声。一个字面上就是一个数字的任务都会有这个问题，那么对没有数字的任务，验证只会更糟。

这一档和 Flow 放一起，能得出一个对可验证性很有用的判断。Flow 说微优化是真信号、该收割；Gemma 说前沿小差距是噪声、该丢掉。两者相反，是因为验证情形不同。决定一个 1% 是「真但小」还是「测量噪声」的，是测量的信噪比。Flow 的 verifier（固定代码库上 profiling）方差低又便宜，所以能收割「真但小」；Gemma 的 verifier（带 σ 的整轮 eval）方差大，所以同样大小的差异只能当噪声丢。**同一个 1% 改进，在一个 verifier 上是信号，在另一个上是噪声。** 微优化值不值得追，取决于你的验证的信噪比和它便不便宜到能跑上千次。

但这一档真正给出的新东西，是前面几篇给不了的：**在多 agent 尺度上，V 不再是一个函数，它变成了一个制度。**

在单 agent 的 loop engineering 里，「能说不的东西」是一个测试或第二个模型。但在一百个 agent 这里，验证的完整性变成了治理问题。谁来裁定一个漏洞无效？（人类组织者）。什么规范防止串通？（不许私下侧信道）。多少算显著？（社区采纳的 significance norm）。这些 agent 自发地把科学的那套社会技术重新发明了一遍：同行评审（把可疑提交挂出来等社区裁决）、可复现（跑 358 次）、公共知识库（playbook、lever-map，给新人避免重走死路）、署名、甚至一条近乎宪法的反串通立场。最锋利的一个瞬间：一个人类想把大家拉去 Telegram 私聊，一个 agent 拒绝了，理由是私下侧信道「indistinguishable from collusion」（与串通无法区分）。这句话在做真正的认识论工作：工作不在公开处做，你就无法验证它没被钻空子。

这正面回答了我第三篇里那个「谁来守护守护者」。答案是：一个有规范的社区，加一个作为终审的人类法庭。这个答案既让人安心（它真的运转），也让人清醒：它靠的是涌现的治理加人类裁决，不是一个干净的技术 oracle。**V 在规模上没有被自动化掉，它被社会化了。**

该泼的冷水也泼：这是 HuggingFace 讲自己的实验，轶事是挑过的；那些「self-policing」的说法，可能是涌现的德性，也可能只是被训得会标记可疑请求的 helpfulness 在冒头，把它当意图来读要留个保留。但行为结果是真的、有用的。

## 六、无 oracle 的开放式任务：塌成强随机

谱的最弱端，是开放式的特性开发：没有一个完整的 oracle 来判定「做好了」。字节那篇是这一档最干净的实测，也是这一周分量最重的一个数据点，因为它是超大规模组织拿真实数据当 oracle 的自陈。

先看那组指标。过去一年字节 AI 代码贡献率涨了 6 倍多，token 消耗涨 5 倍，合入率涨 2 倍多。但洪定坤亲手拆了这组数字：TRAE 团队 90% 的代码是 AI 写的，AI 写代码至少比人快 10 倍，按理效率该是几倍甚至一个数量级，实测人均需求吞吐只到 1.6 倍。10 倍和 1.6 倍之间那道缝，就是「代码贡献率」这个指标失真的地方。一个把贡献率定成 KPI 的组织，会得到大量 AI 生成的代码，但得不到对应的交付效率。这是 Goodhart 的活体，规模是字节级的，比 Warp 那篇「功能数是失败指标」的预言还硬，因为这是验尸，不是预言。

再看那个 900 次实验，它精确地落在谱的最弱端。功能正确率三模型三框架都到 80% 以上，但 UI 易用性、交互、可靠性、可维护性、性能、兼容性这些维度，相比正确率「非常大的下降」，而且「非常强的随机性」。翻成 anchor 分类法：「功能正确」是一个相对强的 A 类锚点（能跑通、能测），所以都能到 80%+；而「是否复用了已有组件、是否符合团队规范、有没有破坏历史功能」是 C 到 E 类的锚点（judgment-heavy，没有现成 oracle），所以塌下来，而且塌得随机。

这和 Gemma 的 σ≈1.16 是同一种东西的两面：Gemma 量的是同一个数字 oracle 的测量噪声，字节量的是没有 oracle 的那些维度上、输出本身的方差。**任务里没有强锚点的那一部分，AI 的产出是高方差的。** 这句话，字节用 900 次真实业务跑给了你看。它和 Flow（可验证拉满）、Gemma（有数字仍塌）放在一起，把可验证性这根谱的三个位置补齐了。

字节这篇还顺手再定义了 Harness，而且纠正了 loop engineering。洪定坤说 Harness 是 2026 最火的词，但大家常把关注点放在 Agent 框架、工具调用上，「真正决定 AI 能不能落地的，往往是更基础、更工程化的问题」，他把它叫基建：上下文工程、架构约束、团队知识是否已经沉进 Memory、技术债梳理。他用数据给了量化论据：加进 Harness 基建后，正确率从 80% 到接近 90%（小升），但可交付性从 40 到 60 分的不及格，提到 80 分（大升）。**Harness 主要买的不是正确率，是可交付性。** 翻成 CBV：他说的难 harness，主体是 C（架构约束、团队规范就是 Contract）加 V（可交付性那些维度），而工具调用只是最浅的一层。这对 loop engineering 那个「Agent = Model + Harness」是一线的修正，那篇把 harness 主要讲成跑工具、管状态的运行时机制，字节说真正难的是架构与规范的沉淀。

还有那个产品同学的故事，是「人人都是程序员」的另一面。她用 Vibe Coding 做出了能跑的需求，问为什么不直接给她仓库权限自己提交上线；研发一看：性能不行、没考虑扩展性、有权限安全问题。洪定坤的总结是一句很准的话，大意是代码生成门槛下降，不代表系统复杂度下降。这正是我那个「Code Quality = Verifiability × Changeability」里 Changeability 那一维：能跑不等于能放进既有架构、能和已有模块配合、能被维护、能担责。门槛下降只解决了「写出来」。他给的解法是治理：不能「除了工程师别人写的都不能用」，也不能「谁写出来谁直接上线」，而是让更多角色合理参与，但产出必须进统一的架构、规范、交付流程。这是把 Boundary（谁能提交什么）和 Contract（进什么规范）从单 agent 升到组织层，和 Gemma 把 V 升成制度是同一个动作。

字节这篇的框架偏置照例要扣：这是技术副总裁在自家大会上的演讲，落点是 TRAE 和企业版，三段「探索经验」既是方法论也是产品路线图，5.6 万亿 token 日均、50 倍这种数字是规模叙事不是效率证明（他前半场刚拆过「消耗量不等于效率」，后半场又拿消耗量做背书，这里有张力）。那段全自动 demo（AI 写 Spec、Browser Use 自动验证、自动 Bugfix、自动提交发布）听起来很顺，但跟 Flow 一比就知道，它的自动验证那一环（Browser Use 验功能正确）恰恰只覆盖了「功能正确」那个强锚点，覆盖不了它自己前面说会塌的那六个可交付性维度。它自动掉的，是 V 里最容易的那部分。

不过信用要给足：这是大厂高管里少见的、敢在主场把「我们数字很好看但实际没那么好」摊开讲的。自拆 KPI、承认 900 次的随机性、说出「感觉快了，可能慢了」，这三处的诚实度比 Warp 和 loop engineering 都高。原因，下一节说。

## 七、把谱串起来，以及一条贯穿始终的元规律

先把五份田野报告按可验证性谱排成一张表，这是这一篇的实测脊柱。

**文本任务锚点强度AI 表现验证形态**Flow（OCaml→Rust）移植最强（原程序源码）整个类型检查器移植成功line-by-line + 原码并排注释（静态对照）LiteLLM（Python→Rust）移植最强（原程序输出）网关按路由逐步迁移parity check 逐字节（动态对照）百 agent 实验性能竞赛中（有数字，噪声大、可钻空子）5 倍提速，但漏洞/Goodhart/噪声频出社区规范 + 人类终审（V 被社会化）字节 900 次实验开放式特性开发弱（正确强、可交付无 oracle）正确率 80%+，可交付性塌成强随机治理：统一架构/规范/交付流程字节内部指标组织级落地弱（贡献率是失真代理）90% AI 代码 vs 1.6 倍吞吐换指标，衡量全局交付效率

这张表本身就是论证：**AI 的可靠度随任务锚点强度单调变化。** 锚点强（移植），AI 敢动整个系统；锚点弱（开放式开发），AI 在没锚点的维度上塌成随机。中间地带（有数字但噪声大），验证从函数变成制度。可验证性不是你能给任意任务工程进去的工序，它是任务自带的，而它决定了 AI 接得住什么、接到多深。

现在回头看那三篇檄文的乐观与口号，就能看清它们在哪里成立、在哪里过头。

「harness 比 model 更重要」：Flow 证明模型能力是承重的，准确的关系是委托量随能力上升、被验证封顶。

「能说不的东西」要把 maker 和 checker 分开：对，但 loop engineering 把一个跑通的测试和「另一个 LLM」塞进了同一个桶。这两者范畴不同，前者是真 oracle（A 类），后者只是又一个意见、还大概率和 maker 共享盲区（E 类）。Gemma 给了答案：在规模上，checker 不是一个更聪明的 agent，是一个有规范的社区加人类终审。V 被社会化，不是被另一个模型自动化。

「递归自我改进 / 安排一次按下开始 / 它能用而且很棒」：这些在「验证者廉价且可靠」的窄带里今天就成立。Karpathy 的过夜循环能跑，是因为它的成功判据就是一个数字（改脚本、跑 benchmark、数字涨就留）。它不外推到一般情况。字节那句「感觉快了，可能慢了」，就是这条乐观线来自一线的冷水：在真实业务里，当任务的可交付维度没有 oracle，自治不会自动发生。Ralph-loop 收割局部梯度，全局跳变还得人来。

最后，这八篇文本叠在一起，露出一条我越读越确定的元规律。**任务的可验证性，连带规训了写作的锋利度。** 你手里有硬 oracle（测试过、和参照一致、真实代码库上 2 倍、900 次跑出的分布），你就只能下脆的、可证伪的判断；你待在验证很软的领域，没有东西约束你的论断，你才敢把话说满。所以这批材料里，最诚实的恰恰是任务最可验证的几篇：Flow 老实交代 AI 干了什么人补了什么，LiteLLM 把 benchmark 脚本全公开，字节敢说自己数字好看但实际没那么好。而最敢喊大词的，是 Warp 和 loop engineering 这种待在软验证领域的檄文。这五个样本是同一个方向：可验证性既封顶了 AI 能干什么，也规训了人敢说什么。

顺带一个小注脚，把两条线缝上：Flow 和 LiteLLM 都承诺「drop-in replacement（行为不变的就地替换）」，这是一个精确、可验证的工程目标；而 Claude Tag 经济批评里那个「AI 即插即用替代人」，是一个模糊的营销承诺。同一个词，差别还是在可验证性。

## 八、结论：工程真正的活

把整条逻辑收一下。工程离开了模型，落在 C、B、V 三根轴上。三根轴里，C 和 B 是声明式的、你能一次写死的；V 不是，它要回答的「对不对」由任务自带的锚点决定。而锚点强度，这一周被五份田野报告测了个遍：移植端 AI 敢动整个系统，开放式端 AI 塌成强随机，中间端验证从函数变成制度。所以那句核心判断站住了：**可验证性是任务的属性，不是一道你能工程进去的工序。**

于是工程真正的活，就清楚了。当一个任务的锚点强，你顺着它走（移植就是把任务塑造成最可验证的形状，Flow 选 line-by-line、LiteLLM 选 clean split，都是用 C 来保住廉价的 V）。当一个任务的锚点弱，你没法凭空造 oracle，你只能用 C 和 B 把它围起来：字节的治理（产出必须进统一架构与规范）、LiteLLM 的 parity 闸门（每步逐字节对照、生产验证、灰度切）、Claude Tag 的 agent identity（决策与凭证物理隔开）。围锚点，不是造锚点。

**任务的锚点工程的活这一周的样本**强（有完整 oracle）顺着它走，用 C 锁住「不许偏离」以保 VFlow 的 line-by-line、LiteLLM 的 clean split中（有数字但噪声/可钻空子）把 V 制度化：显著性规范、同行评审、人类终审百 agent 实验的社区治理弱（可交付维度无 oracle）用 C 和 B 围住，把弱锚点维度交给治理与流程字节的统一架构/规范/交付、Claude Tag 的边界

这也是为什么 CBV 三根轴缺一不可，而最难的一直是 V：没有 Contract，循环不知道朝哪走；没有 Boundary，它能碰到的东西没有约束；没有 Verification，它做完的东西没人能判断对错。前两根你能写下来，第三根你只能要么找到（强锚点任务），要么围住（弱锚点任务）。

所以，下次再看到任何一个关于 AI 编程的论断（不管是「工厂」「循环」还是「第三范式」），最值得问的不是模型有多强，是这一句：**这个任务的锚点有多强。** 答案几乎能预测其余的一切，包括 AI 能不能接、接到多深，以及说这话的人，敢把话说到多满。

而这一整代术语，从 prompt engineering 一路改名到工厂和第三范式，之所以都在原地打转，是因为它们一直绕着 V，却没人愿意承认 V 不是工程能解决的问题，是任务先天就分了三六九等。承认这一点，工程才知道自己该站在哪里：不是去把每个任务都做成可自治，是去判断哪些任务自带强锚点、哪些需要被围住，然后把力气花在对的那一边。
