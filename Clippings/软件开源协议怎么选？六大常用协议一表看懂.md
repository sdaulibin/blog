如果你也在做开源项目，但不知道应该选哪个开源协议，下面六个覆盖 95% 的场景。帮你快速选型👇

## 一、总览对比

| 协议 | 宽松度 | 能否闭源 | 专利保护 | Copyleft 强度 | 代表项目 |
| --- | --- | --- | --- | --- | --- |
| MIT | 极高 | 可以 | 无 | 无 | React、Node.js、Vue.js |
| Apache 2.0 | 极高 | 可以 | 有 | 无 | Kubernetes、TensorFlow、Swift |
| BSD | 极高 | 可以 | 无 | 无 | FreeBSD、OpenBSD |
| MPL 2.0 | 中 | 部分可以 | 有 | 弱（按文件） | Firefox、Thunderbird |
| GPL | 低 | 不可以 | v3 有 | 强（整个项目） | Linux、WordPress、GCC |
| AGPL v3 | 低 | 不可以（含云服务） | 有 | 最强（含网络） | Nextcloud、Bitwarden |

前三个是**宽松型**（最大化传播，别人闭源赚钱你管不着）；后三个是 **Copyleft**（用了我的，你也得开源）。

## 二、逐项解释

| 协议         | 一句话本质         | 唯一要点                                           |
| ---------- | ------------- | ---------------------------------------------- |
| MIT        | 最佛系的"随便用"     | 只要求保留版权声明，其余全放行。想火、想被大厂集成就选它。                  |
| Apache 2.0 | MIT + 专利授权    | 多一层防专利反咬的保护。做企业级、AI、大数据用它，别用 MIT。              |
| BSD        | MIT 的双胞胎      | 几乎和 MIT 一样；3-Clause 版仅多一条"不准用作者名做宣传"。常见于系统/科研。 |
| MPL 2.0    | 最聪明的中间路线      | 只有  改过的文件  要开源，其余可混闭源代码。既护核心又留商业空间。            |
| GPL        | 强 Copyleft 铁律 | 碰了代码，整个衍生项目都得开源，禁止闭源商用。v3 额外防硬件锁。              |
| AGPL v3    | 云时代防白嫖核武器     | 堵住 GPL 漏洞：只要对外提供网络服务，改动也必须公开。SaaS 必备。          |

## 三、选型决策

| 开发者类型 | 典型诉求 | 推荐协议 |
| --- | --- | --- |
| 个人 / 独立开发者 | 攒口碑、求采用率、做作品集 | MIT |
| 库 / 框架作者 | 让下游无门槛集成 | MIT / Apache 2.0 |
| 大厂 / 企业开源团队 | 合规 + 专利安全 | Apache 2.0 |
| SaaS / 云服务创业者 | 开源引流又防白嫖 | AGPL v3 |
| Open Core 商业化团队 | 核心开源、增值闭源 | MPL 2.0 / AGPL v3 |
| 社区 / 理想主义维护者 | 守住自由、防大厂闭源 | GPL |
| 高校 / 科研人员 | 最大化传播、零负担 | BSD / MIT |

## 四、三个坑

| 坑 | 后果 |
| --- | --- |
| 以为协议能随便改 | 改协议要全体贡献者同意，项目越大越改不动，定了基本回不了头。 |
| 做 SaaS 却用 GPL | 留了大漏洞，别人改完只在云端跑就能合法白嫖，认准 AGPL。 |
| 嫌麻烦不写 LICENSE | 默认"保留所有权利"，别人法律上根本不能用，等于白开源。 |
