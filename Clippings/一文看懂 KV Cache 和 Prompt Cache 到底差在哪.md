下面这张 DeepSeek 的定价图，很多人应该都见过。

![图像](https://pbs.twimg.com/media/HIqq2qZbcAANHPl?format=jpg&name=large)

看一眼里面这两栏数字：

- 缓存命中：0.025 元 / 百万 tokens
- 缓存未命中：3 元 / 百万 tokens

同一个模型，同一段输入，差 **120 倍**。

那么缓存究竟是什么呢？为什么能够让模型价格差这么多，然后每个模型厂商都在强调缓存要怎么做，把缓存命中率作为生命线来维护。

这篇就来把这件事讲清楚：什么是缓存，KV Cache 和 Prompt Cache 分别是什么，怎么让自己的请求多命中缓存，日常用的中转站里这些字段又该怎么看。

但聊缓存之前，得先聊一个绕不开的词：Token。

## 先聊聊 token

所有 AI 公司都按 token 收费，不是按问题数，也不是按字数。原因很简单：模型眼里没有字、没有问题、没有句子，只有 token。

token 是模型把人类语言切成的最小处理单元。但同一段话切出来多少 token，不同模型给的答案不太一样。OpenAI 在自家 cookbook 里明确说过：不同模型用不同的 encoding，同一段字符串在不同 tokenizer 下切出来的 token 数能差挺多。

要个粗略概念，可以看 DeepSeek 官方给的口径：1 个中文字符大约 **0.6 token**，1 个英文字符大约 **0.3 token**。但 DeepSeek 自己也强调，这只是粗估，实际数量取决于模型 tokenizer，**以 API 返回的 usage 为准**。

Anthropic 那边没公开"中文每字多少 token"这种固定比例，只给了一个 token counting 接口让你自己去数，文档里也明说返回的是估算值。

为什么按 token 计费？因为模型每处理一个 token 都要做一遍矩阵运算，所有显存、算力、电费，最终都按 token 这个粒度被消耗。按字数收费不准，不同 tokenizer 切出来的 token 数本来就不一样。按问题收费更离谱，一个"你好"才 2 个 token，一份几万字的财报丢进去就是几万 token，算力差几千倍。

所以下面讲的缓存，讲来讲去都是同一件事：哪些 token 是新算的，哪些 token 是从缓存里直接拿的。前者按原价收，后者按打折价收，两者中间能差一两个数量级。

## 一. 没有缓存的 AI 有多蠢

先记住一个反直觉的事实：模型每蹦一个新字，都要把前面所有字重新算一遍。

LLM 的工作方式叫自回归。AI 圈管这个叫自回归，翻译过来就是**单字蹦迪**——一个字一个字往外蹦，蹦下一个字的时候必须看见前面所有的字。

如果不做任何优化，这件事会变成灾难。

你让模型续写一篇 1 万字的文章。生成第 10001 个字时，模型要把前面 1 万个字全部塞进注意力机制重算一遍。生成第 10002 个字，再算一遍。生成第 10003 个字，又算一遍。

更可怕的是，这个计算量不是线性增长。文章长度翻倍，计算量不是翻倍，是翻 4 倍。10 万字的文章生成下一个字，理论上要做的乘法次数，是 1 万字的 100 倍。

这就是注意力机制的二次方爆炸。字越长，算力消耗呈指数级膨胀。

放到真实模型上有多吓人？Llama 3-70B 想撑住 100 万 token 的上下文，光是中间过程产生的临时数据，就要占掉 330 GB 显存。一张顶级 H100 也才 80 GB，根本装不下。在没有任何优化的年代，长上下文这件事在工程上是不可能的，更别说实时生成了。

我之前看到一个比喻特别贴切：一个学生读小说，每读到一句新句子，都要从第一页第一个字开始重新读一遍才能理解。读到第 100 页时，他实际已经把第 1 页读了 100 遍，把第 50 页读了 50 遍。

显然没人会这么读书。工程师们也不能让模型这么干，不然 GPU 早烧穿了。

所以模型必须想办法记笔记。记下来的笔记，就是后面要讲的缓存。

## 二. KV Cache：阅读时的草稿纸

最早被工程师发明出来记笔记的工具，叫 **KV Cache**。

要明白 KV Cache 在干什么，得先回到模型工作的本质：**模型就是一个根据前面的字预测下一个字的机器**。

举个例子。你给它输入 今天天气真，模型读完这 5 个字，预测出下一个字大概率是 好。把 好 拼上去，输入变成 今天天气真好，模型再去预测第 7 个字，可能是逗号，也可能是 啊。再拼上去，再预测第 8 个字，循环往复。

问题来了：预测第 7 个字的时候，前 6 个字在注意力层算出来的那堆 K 和 V，跟刚才算第 6 个字时是一模一样的。如果每生成一个字都把前面所有字重算一遍，就是上一节讲的二次方爆炸。

![图像](https://pbs.twimg.com/media/HIqq8wwaUAAKsky?format=jpg&name=large)

KV Cache 干的事就是：**把前面已经算过的 K 和 V 存在显存里，下一个字只算自己的那一份，前面那些直接拿来用。**

K 是 Key，可以理解为目录索引；V 是 Value，是索引指向的具体内容。每个 token 在每一层注意力里都会算出这两份东西。

![图像](https://pbs.twimg.com/media/HIqq_H7aAAAXtm8?format=jpg&name=large)

打个更日常的比方。做阅读理解时你会在草稿纸上记下核心人物、关键事件、重要数字。读第二段不用回头重读第一段，直接看草稿纸就能续上。KV Cache 就是这张草稿纸。

有了 KV Cache，那个二次方爆炸的灾难就被压平了。模型生成第 10001 个字，只需要算这一个字自己的 K 和 V，再去查前面 1 万个字的草稿纸。算力消耗从指数级降到了线性。

![图像](https://pbs.twimg.com/media/HIqrMhyboAE09P_?format=jpg&name=large)

但 KV Cache 有三个关键限制，记住这三条后面才好理解 Prompt Cache：

- **只在一次请求里有效**。这次对话生成完，整张草稿纸直接撕掉
- **不影响 API 计费**。你输入 1 万 token，模型该收多少还是收多少。KV Cache 省的是模型自己的算力，不是你的钱
- **所有 LLM 都有**。GPT、Claude、Gemini、DeepSeek、通义、文心，没有 KV Cache 的模型今天已经跑不起来了

KV Cache 是大厂为了不让自己破产必须做的底层基建。机器要能跑得动，KV Cache 是起点不是终点。

留个伏笔：每来一个新请求，草稿纸都要从头重画。一万个用户问同一个 system prompt 下的问题，就要画一万张几乎一样的草稿纸。这不浪费吗？

为了解决这件事，工程师们造了第二样东西。

## 三. Prompt Cache 和 DeepSeek 的硬盘缓存

回到那个一万用户的场景。如果都用同一个 system prompt，前面 5000 个 token 完全一样，谁来重算 9999 次都觉得心疼。

**Prompt Cache** 干的就是这件事：跨请求复用，一个用户算过的前缀，下个用户直接拿。

要讲清楚 Prompt Cache 和 KV Cache 的区别，得先把模型干活的过程切成两半。

模型处理一次请求分两步。第一步叫 **prefill**，把你输入的整段 prompt 读进来，算出每个 token 的 K 和 V。第二步叫 **decode**，一个字一个字往外蹦回答。KV Cache 省的是 decode 那一步的算力，Prompt Cache 省的是 prefill 这一步的钱。

一个用来加速生成，一个用来加速读题。两者作用阶段不同，作用域也不同。

![图像](https://pbs.twimg.com/media/HIqrQjuagAETJEU?format=jpg&name=large)

维度 KV Cache Prompt Cache 作用范围 单次请求内 跨请求共享 影响计费 不影响 直接打折，最高省 90% 角色定位 底层基建 商业产品

KV Cache 是大厂为了不让自己破产必须做的底层基建，Prompt Cache 是大厂为了抢客户主动推出的商业产品。一个是技术兜底，一个是降价利器。

![图像](https://pbs.twimg.com/media/HIqrTmKa4AAwUga?format=jpg&name=large)

**DeepSeek 为什么能比别家便宜这么多**

OpenAI 和 Anthropic 的 Prompt Cache 都放在显存里。显存贵又少，留不住东西，5 到 10 分钟就被挤掉了。Anthropic 把这套机制直接写进了定价：以 Claude Opus 4 系列为例，原价 $5 / M tokens，命中缓存读取只要 $0.50 / M，相当于打 1 折（5 分钟和 1 小时寿命的写入分别要 $6.25 / M 和 $10 / M）。

DeepSeek 干了一件别家没做的事：**把 Prompt Cache 直接写到硬盘上**，官方叫 **Context Caching on Disk**。

最容易误解的点：它不是让模型每生成一个字都去硬盘读 KV，那真会拖死。它存的是 prompt 前缀的 prefill 结果（算好的 K/V 状态），下次同样开头进来，prefill 直接跳过，把状态从硬盘加载进显存，再正常 decode。

省下的是更贵的那一步：一段几十万 token 的 prefill 算力，远比 NVMe 读一次缓存的 I/O 成本贵。前提是缓存体积要够小，DeepSeek 用自研的 **MLA**（多头潜在注意力）把 KV Cache 压到了极小，刚好能塞进硬盘阵列。

效果是寿命直接拉满：别家显存里 5 到 10 分钟，DeepSeek 硬盘里几小时到几天（官方说是 best effort，不保证 100% 命中）。你今天发的请求，前缀可能命中了昨天某个陌生人发过的同一段话。两个素不相识的人，因为开头写得一样，第二个人直接享受了第一个人付过的算力账单。

这才是 120 倍价差的真正秘密。不是模型小，是缓存做到了硬盘上。DeepSeek 还把它做成默认开启，每次响应里直接返回 prompt\_cache\_hit\_tokens 和 prompt\_cache\_miss\_tokens，命中多少一目了然。

## 四. Claude Code 团队的 6 条 cache 铁律

知道了原理，下一个问题是：怎么写 prompt 才能让缓存命中率最高。

先看一组真实数据。

技术专家 Ankit Sinha 做过一个严谨的统计实验。优化前缀的请求平均花 **$0.009556**，故意在开头加随机干扰字符破坏前缀的请求平均花 **$0.033306**。在几千次请求的对照实验里，优化组总花 $6.83，对照组 $19.48。**P 值小于 0.000001，t 统计量 -8.65**，统计上完全排除偶然性。

把这个数字推到企业生产线。每天万级请求的规模下，仅仅因为开头加了一个不必要的时间戳，企业的年度 API 账单从 **9.5 万美元飙到 33.3 万美元，多烧近 25 万**。一个时间戳的代价。

这不是夸张。Anthropic 自己的 Claude Code 团队在公开复盘里直接说，他们整个工程架构都是围绕 prompt caching 设计的，命中率掉一点都当 SEV 报。下面是他们公开的 6 条铁律。

1\. 静态在前，动态在后

Claude Code 的 prompt 严格分 4 层，从前到后依次是：

- 第 1 层 **Static system prompt + Tools**（全局共享缓存）
- 第 2 层 **Claude.MD**（项目级缓存）
- 第 3 层 **Session context**（会话级缓存）
- 第 4 层 **Conversation messages**（动态对话）

越靠前的越稳定，越靠后的越多变。这个顺序保证了不管哪个会话进来，前 3 层都能命中缓存。

![图像](https://pbs.twimg.com/media/HIqrWMVaYAA6PXV?format=jpg&name=large)

Claude Code 自己也踩过坑：在 static system prompt 里塞过详细时间戳，结果每分钟缓存失效一次；工具定义顺序非确定性洗牌，每次都换位置，缓存当场全废。

2\. 要传更新信息？走 message 通道

日期变了、用户改了文件、当前选中的代码不一样了，这些更新信息别去改 system prompt。

system prompt 一改，前缀就变，整段缓存全废。

Claude Code 的做法是：在下一条 user message 或者 tool result 里加一个 <system-reminder> 标签，把更新信息塞进去（比如 **现在是星期三** 这种）。新的信息照样送达模型，但前缀稳定不变，缓存继续命中。

3\. 不要中途换模型

这条最反直觉。

假设你跟 Opus 聊了 10 万 token，问到一个简单问题，想切到 Haiku 省点钱。直觉上是省了，实际更贵。

因为缓存是绑模型的。从 Opus 切到 Haiku，等于要给 Haiku 重新建一遍 10 万 token 的缓存。这一笔 prefill 算下来，比 Opus 把这个问题答完还贵。

要切模型怎么办？用 subagent。让 Opus 准备一个 handoff message，把上下文压缩成几百 token 交给 Haiku 处理。Claude Code 自己的 Explore agent 就是这么用 Haiku 的。

4\. 不要中途加减工具

工具列表也是 cached prefix 的一部分。加一个工具、减一个工具，整段对话的缓存立刻失效。

这条最容易踩的场景是 **plan 模式**。

直觉上，进 plan 模式应该切到只读工具集（不能让 agent 改文件）。但这一切就破坏缓存。Claude Code 的做法很精妙：把 EnterPlanMode 和 ExitPlanMode 设计成两个工具本身，所有工具永远在线，只通过这两个工具触发状态切换。

附带好处：agent 自己也可以在发现难题时主动调 EnterPlanMode 进 plan 模式，全程缓存不中断。

5\. 工具多就 defer\_loading，不要删

MCP 工具几十个全塞进 prompt 里，每次请求带的 schema 都贵到肉疼。删掉一些？又破坏缓存。

Claude Code 的解法叫 **Tool Search**。发出去的工具列表里，不重要的那些只放一个轻量 stub，只有工具名和 defer\_loading: true。模型在需要时通过一个叫 ToolSearch 的工具调出完整工具，schema 只在选中时才加载。

这样，cached prefix 永远稳定，所有 stub 顺序固定，工具数量恒定，缓存命中率拉满。

6\. 像监控 uptime 一样监控缓存命中率

Claude Code 对缓存命中率设了告警阈值。命中率掉到某个数以下，工程师团队当 SEV 报。

这是他们公开的运维红线。把缓存命中率当 SLA 来管，跟把服务可用性当 SLA 来管，一样严肃。

把这 6 条归到一处，讲的都是同一件事——**前缀守住**。任何一个字符的变化，都会让后面全部作废。

## 五. 怎么在中转站看缓存读写

讲了这么多原理和规则，你怎么知道自己写的 prompt 真的命中了缓存？

光看 API 文档不够。命中没命中，得去日志里看。

最直接的地方是中转站的使用记录。我自己运营的 Dragon Code，底层是基于 Sub2API 这套系统。市面上主流的中转站基本都是 Sub2API 或 NewAPI 同源，字段长得都差不多。

打开一次请求的明细，你会看到四个关键字段：

- **↓ 绿色**：输入 token。这部分是没命中缓存的，按原价算
- **↑ 红色**：输出 token
- **📦** **蓝色**：缓存读取 token。命中了缓存，按打折价算
- **✏️** **黄色**：缓存写入 token。首次创建缓存有一点溢价

举个真实的例子。我前几天有一次请求长这样：

- 输入成本 $0.000005（只有 1 个 token 走原价）
- 缓存读取成本 $0.095548（90.6K token 命中缓存）
- 缓存创建成本 $0.008081（3.1K token 写入新缓存）
- 总计 $0.103634

如果这次请求全部走原价（93.7K token 按普通输入价），账单大概是 $0.453。也就是说，写得 cache-friendly 之后，这一次请求省了 **4.7 倍**。

NewAPI 体系的中转站也是一样，输入日志里都能看到对应的缓存读写字段。

![图像](https://pbs.twimg.com/media/HIqraDVaIAAajql?format=jpg&name=large)

这里说一下三种最容易踩的坑，看上去命中、其实没命中：

1. **首次请求不会命中**。你的 prompt 第一次发出去，正在创建缓存，看到 ✏️ 黄色那栏是正常的，不要慌
2. **不同账号、不同 API Key 之间不共享**。OpenAI 和 Anthropic 文档里都明确写了，prompt cache 不跨组织。两个团队即使用一模一样的 system prompt，缓存也是各算各的
3. **中转站偷偷改了 system prompt**。一些中转站会注入自家的安全提示词，或者在尾部追加用户 ID。这种操作会把前缀污染掉，永远命不中。挑中转站时这一条比价格还重要

把这三条记下来。下次看到自己的账单贵得离谱，先去日志里看一眼是不是缓存没命中，而不是怀疑模型涨价了。

## 写在最后

回到开头那张表：0.025 元 vs 3 元。

同样是 100 万 token，价格能差到 120 倍，靠的不是一句神秘咒语，就是缓存命中了。

下次再有人问你，LLM 的缓存到底是什么，你可以这么讲：

**KV Cache 是模型自己的草稿纸，让它在同一次生成里别重复算前文。Prompt Cache 是服务商的公共草稿纸，让多个请求复用同一段开头。**

真要记一句话，就记这个：**静态在前，动态在后。**

以后写 system prompt、工具定义、长文档，尽量放前面，别天天变。用户问题、时间、随机 ID，尽量放后面。

AI 价格战打到最后，拼的不只是模型能力，也是谁更会把已经算过的东西留下来。对开发者来说，理解这一点，可能比多换几个便宜模型更有用。
