我从工程师转做出海业务，这段时间里，把 AI 一点点塞进团队日常工作里，踩过的坑不少。回头想想，有许多深刻的体会：团队里那些反复出问题的地方，十有八九不是谁能力不行，而是该说清楚的规矩，从来没人好好写下来。

CLAUDE.md 这东西，其实就是这样的事情在 AI 编程工具上的具象化。

讲个场景，很多团队都会遇到。

团队里只要有第二个人在同一个仓库上用 Claude Code，那么：同一个项目，Claude 在每个人的电脑上，表现得一定是不同的人。

在你这里，它老老实实写测试、按你们的规范来、改动小心翼翼。换到同事那台机器，它能把整个组件推倒重写，顺手生成一堆没人想看的文档，代码风格还是它自己现编的。再到新来的实习生那边，它压根不知道你们的接口该返回什么结构。

谁都没做错，问题是给 Claude 定的所有"规矩"，全散在两个永远不会同步的地方：每个人的脑子里，和每个人本地那台电脑里。

这里拿一个例子进行讲解：一个5 个人的做跨境电商的出海团队，前后端分离的 monorepo，主要服务东南亚。下面所有配置、所有困难，都在这个项目里发生。

## 一、分层：三个文件

很多人对 CLAUDE.md 的第一个误会，是以为它就一个文件。

它不是。Claude Code 启动的时候，会一口气读好几个文件，然后揉成一份完整的指令。

具体怎么读？它从你当前所在的目录出发，顺着目录树往上爬，一直爬到 git 根目录，沿途碰到的每个 CLAUDE.md 都收进来；同时它还会去你的 home 目录里，把那个全局配置也拿上。这些文件叠在一起，才是 Claude 真正看到的规则。

有三层需要分清楚：

```text
~/.claude/CLAUDE.md     →  个人级。跟着你这个人走，你所有项目都受它影响。永远不进 git。
./CLAUDE.md             →  项目级。提交进 git，全队共用。这是团队的"宪法"。
./CLAUDE.local.md       →  你对当前这个项目的私人覆盖。会自动进 .gitignore，不入库。
```

它们怎么合并？就一条：越具体的，越说了算。同一条指令在好几个文件里冒出来、还互相矛盾的时候，更具体的那个优先。具体程度从高到低排：CLAUDE.local.md 最高，然后 ./CLAUDE.md，最后才是 ~/.claude/CLAUDE.md。

拿出海团队那个例子，实打实演示一遍。

团队里有个工程师叫小李。他那三个文件，长这样：

他自己机器上的个人级 ~/.claude/CLAUDE.md：

```markdown
- 回答我时用中文
- 给我讲代码尽量讲细点，我想知道为什么这么做
- 默认用 4 空格缩进
```

项目级 ./CLAUDE.md，全队共用、早就进了 git：

```markdown
- 本项目统一 2 空格缩进
- 所有 API 调用必须包 try/catch
- 数据库查询必须用 projection
```

还有他的 ./CLAUDE.local.md，只在他电脑上，没进 git：

```markdown
- 我负责支付模块，要动 src/payments/ 以外的代码，先问我
```

那小李在这个项目里用 Claude Code，Claude 最后听的，是这么一份合并出来的结果：

- 中文回答（个人级来的，没人覆盖）
- 讲解讲细点（个人级来的）
- 缩进用 2 空格——这里要停一下。个人级说 4 空格，项目级说 2 空格，两边打架。项目级更具体，赢。小李的个人习惯，在这个项目里被盖掉了。这是对的，团队一致优先。
- API 调用包 try/catch（项目级）
- 数据库查询用 projection（项目级）
- 动支付模块以外的代码先问他（他自己的 local）

这就是"合并"真实的样子。

看懂这个，你就抓住了最关键的一点：一条规则该放哪一层，你只要问自己一句话：这是团队的决定，还是我个人的偏好？

技术栈、构建命令、编码规范、API 约定、PR 规则，这些是团队决定，进项目级。你用什么语言跟 Claude 说话、喜欢它讲多细、你本地数据库开在哪个端口、你负责哪个模块，这些是个人偏好，进个人级或 local。

放错层会怎样？规则就"漏"到不该去的地方。举个最常见的：有人手一抖，把"用 4 空格"这种纯个人习惯写进了项目级 CLAUDE.md——好了，全队的 Claude 都被他一个人的习惯带跑了。文章开头那个"Claude 像不同的人"的怪象，很多时候就是这么来的。

## 二、CLAUDE.md 是命令

模板先别急着抄。有个特别普遍的写法错误要改，不然模板抄回去也写不对。

写 CLAUDE.md的时候，很容易写着写着就跑偏成了项目介绍——"本项目是一个面向东南亚市场的跨境电商平台，采用前后端分离架构……"。

打住。这是 README 的口气。CLAUDE.md 不该这么写。

记住一件事：CLAUDE.md 会被整段塞进 Claude 的系统提示里。它里面的每句话，都得是一条能直接照做的命令，不是一段给人看着舒服的介绍。

感受一下这组对比。

读者经常写成这样，描述式：

```markdown
我们项目比较重视类型安全，大家平时写代码
应该尽量注意类型问题，TypeScript 用得比较严格。
```

而 CLAUDE.md 真正该有的样子，命令式：

```markdown
- TypeScript 严格模式，禁止用 \`any\`
- API 出入参必须声明为带类型的常量
```

差在哪里？上面那段，"比较重视""尽量注意""用得比较严格"——全是软绵绵的形容词，Claude 读完没法做任何明确判断。下面这两条，"禁止""必须"，边界硬邦邦的，一点不含糊。

写的时候，你脑子里不该想着"我在介绍我的项目"。你该想的是：我面前坐着一个能力很强、但对我们团队规矩一无所知的新同事，我要给他列一份没得商量的规矩。多用祈使句。多用"必须、禁止、总是、绝不"。少用"建议、尽量、一般来说"。

## 三、能直接使用的模板

上面两点想明白了，这份模板才有意义。放仓库根目录，叫 CLAUDE.md，提交进 git。这就是那个出海团队的"宪法"。

```markdown
# CLAUDE.md

## 项目概览
跨境电商平台，服务东南亚市场。前后端分离的 monorepo。

## 常用命令
- \`pnpm dev\`        启动开发服务器
- \`pnpm build\`      生产构建
- \`pnpm test\`       运行全部测试
- \`pnpm lint-fix\`   Lint 并自动修复所有包

## 项目结构
- \`packages/client\`  前端（Vue 3）
- \`packages/server\`  后端 API（Express）
- \`packages/shared\`  共享类型、DTO、工具函数

## 编码规范
- TypeScript 严格模式，禁止用 \`any\`
- 只用绝对路径导入，禁止 \`../../\` 这种相对路径
- API 出入参声明为带类型的常量
- 统一 2 空格缩进

## API 约定
- 所有接口返回统一信封：{ code, data, message }
- 错误码集中定义在 packages/shared/error-codes.ts
- 分页参数统一用 page / pageSize，禁止用 offset / limit
- 时间字段一律返回 ISO 8601 字符串，禁止返回时间戳

## PR 审查规则
1. 类型安全：绝不用 \`any\`
2. 死代码：删掉没用的 import、变量、i18n key
3. 错误处理：所有 API 调用必须包 try/catch

## 关键陷阱（团队踩过的坑）
- 数据库查询必须用 projection，绝不要整文档拉取
- 服务端报错信息必须来自常量文件，禁止硬编码字符串
- 涉及金额的计算必须用整数（分），禁止用浮点数

## 工作流
- 从 main 切 feature 分支
- 所有 PR 用 squash merge
- 合并后删分支
```

这模板为什么这么排？说实话，这些"为什么"比模板本身值钱？

"常用命令"为什么要列出来？因为 Claude 不知道你们用 pnpm 还是 npm，也不知道你们的测试命令叫啥。你不写，它就猜。猜错了，命令就跑错了。把团队天天敲的那几条写下来，它要用的时候才能直接用对。

"关键陷阱"这一节，个人觉得它的价值压过"编码规范"。编码规范那些东西是通用知识，Claude 本来就懂个七七八八。可"金额必须用整数分""报错信息必须来自常量文件"——这是你们团队拿事故换来的、独一份的经验，Claude 再聪明也猜不到。一份 CLAUDE.md 里最值钱的，永远是这种"只有你们才知道"的部分。要是你的 CLAUDE.md 通篇都是"要写注释""要有可读性"这种正确的废话，那它基本等于没写。

还有长度。一份好的 CLAUDE.md，尽量压在 200 行以内。为什么？它会被整个塞进系统提示。你每多塞一条没什么价值的指令，那些真正关键的指令权重就被稀释一分。当文件里堆满"代码要优雅"这种谁都同意、可谁也没法照着做的话，Claude 反而更难抓住"金额用整数分"这种真正的硬规矩。所以宁可少，别贪多。

再举一个反面教材，照照镜子。下面这份，就是典型的烂 CLAUDE.md：

```markdown
# CLAUDE.md
我们是一个很棒的团队，做的是电商项目。
代码质量很重要，大家写代码注意一下质量。
要写得优雅点，有可读性，注意性能。
最好多写点注释，测试也尽量补补。
```

烂在哪？全是形容词，没一条能照着做；全是放之四海皆准的废话，没一条是这个团队特有的。Claude 读完它，对这个项目的了解，跟读之前一模一样。

这里有两条硬性提醒：

一、别往里放敏感信息。API key、密钥、数据库连接串、详细的安全漏洞细节，统统别放。这文件会进系统提示，还会进 git。当它是一份随时可能被公开的文档来对待。

二、可以用 /init 生成初稿，但记住它只是初稿。在项目里跑 /init，Claude 会扫一遍你的代码库，自动吐一份 CLAUDE.md 出来，构建命令、关键目录、它检测到的代码风格，通常都能识别。可它抓不到你们的隐性约定，就是那些"金额用整数分"的坑。把 /init 的产出当起点，然后人工把它不可能知道的东西补上。

## 四、把项目 CLAUDE.md 当代码管

项目级 CLAUDE.md 是代码。它的每一次改动，都得走 Pull Request，走 Code Review。

这文件影响所有人，你改一个字，全队五个人的 Claude 行为同时变。这跟你改一个被到处调用的公共函数，风险一个量级。你不会让谁不经 review 就直接 push 改公共函数吧？那你也不该让谁直接 push 改 CLAUDE.md。

你要是放任大家随便改，等着你的是这种循环：工程师 A 加了条"函数必须分号结尾"，工程师 B 嫌烦，下回顺手删了；有人调试时把自己本地的环境路径写进去，完事忘删；下一个新人 clone 仓库，Claude 一开局就跟全队都不一样。规则没人认领，就是这个下场。

走 PR，具体长啥样？

改 CLAUDE.md 的 PR，描述里得说清三件事：改了啥、为啥改、对全队啥影响。还是那个出海团队，某次的 PR 描述可以这样写：

```text
标题：CLAUDE.md：新增「金额必须用整数分」规则

改了啥：
「关键陷阱」一节加一条 —— 涉及金额的计算必须用整数（分），禁止浮点数。

为啥改：
上周印尼站结算冒出 0.01 的金额误差，查了大半天，定位到一处用 float 累加金额的代码。
根因是浮点精度。这坑会反复踩，得固化成规则。

影响：
往后 Claude 写任何涉及金额的代码，默认都用整数分。
现有的浮点金额代码不在这次改动范围，另开 issue 跟。
```

那 review 这条 PR 的时候，reviewer 看什么？不看代码，看这条"规则"本身合不合格：

- 它是命令，还是描述？（口气够不够硬）
- 它是这个团队独有的，还是通用废话？（配不配占这一行）
- 它会不会跟文件里已经有的某条规则打架？
- 它该进项目级，还是其实是某个人的偏好、该塞进 local？

往深了说一句：团队一旦开始用 CLAUDE.md，中文也好英文也好，你们的自然语言就变成了一种"元编程语言"。你用它编程的对象，不是机器，是 AI 的行为。既然是编程，它就该享受跟代码一样的待遇：版本管理、Code Review、改动留痕。一样都不能少。

## 五、规则之间打起来了怎么办

CLAUDE.md 越写越长，规则互相矛盾这件事，迟早碰上。分两种，有两种处理方式：

第一种，跨层级冲突。个人级说"4 空格"，项目级说"2 空格"。这种好办，有明确的裁决规则：越具体越优先，项目级优先。这不是 bug，是设计本身，第一节已经讲过了。

第二种麻烦点，同层级冲突。同一个项目级 CLAUDE.md 里，A 早先加了"所有函数都要写单元测试"，B 后来又加了"原型阶段的代码先别写测试"。这俩在同一层，谁也不比谁更具体。Claude 撞上，只能自己猜该听哪条。猜的结果就是行为飘忽不定。

同层级冲突，没有自动裁决，只能靠人。这恰好就是"把 CLAUDE.md 当代码、走 PR review"的价值所在。B 提交"原型阶段不写测试"那条的时候，reviewer 但凡尽职，就该一眼看出它跟已有的"所有函数都要写测试"打架，然后让 B 在 PR 里把规则改清楚。比如改成这样："packages/server 下的代码必须写单元测试；experiments/ 目录下的原型代码可以不写。"

你看，矛盾被一条更精确、带了路径范围的规则给化掉了。这就是为什么 review 这关不能省，同层级冲突，它是唯一的拦截点。

## 六、用 .claude/rules/ 目录拆开

团队规则越攒越多，全堆在一个 CLAUDE.md 里，文件臃肿到没人想读的时候，就该拆了。

啥时候算"该拆"？给个粗糙的判断线：CLAUDE.md 超过 150 到 200 行，或者团队里开始有不同的人分管不同子系统（前端、后端、数据）的时候，就值得拆。5 个人以下、就一个模块的小项目，一个文件够用，别瞎折腾。

拆，是拆进 .claude/rules/ 目录。结构这样：

```text
.claude/
  rules/
    api.md          # API 设计约定
    components.md   # 前端组件规范
    testing.md      # 测试规则
    database.md     # 数据库查询规范
CLAUDE.md
```

然后在主 CLAUDE.md 里用 @ 语法把它们引进来：

```markdown
# CLAUDE.md

## 项目概览
跨境电商平台，服务东南亚市场。前后端分离的 monorepo。

## 常用命令
- \`pnpm dev\`     启动开发服务器
- \`pnpm test\`    运行全部测试

## 团队规范（详见各 rules 文件）
@.claude/rules/api.md
@.claude/rules/components.md
@.claude/rules/testing.md
@.claude/rules/database.md
```

被引进来的文件，比如 .claude/rules/database.md，就专心写它那一块：

```markdown
# 数据库查询规范

- 所有查询必须用 projection，只取要的字段，禁止整文档拉取
- 列表查询必须分页，单次返回上限 100 条
- 涉及金额的字段以整数（分）存储和读取
- 跨表关联优先在应用层做，禁止超过三层的嵌套 join
```

拆完有两个好处。一个比较表面：文件短了、好读了、改动时 diff 也清爽。另一个才是重点：拆完之后，规则有了明确的主人。谁管 API 层，谁维护 rules/api.md；前端负责人管 rules/components.md；数据那块谁负责，谁管 rules/database.md。规则的归属，跟代码子系统的归属对齐了。这些文件全部进 git，因为它们代表的是团队决定，不是个人偏好。

补一句实用的：@ 引用这套语法，在 Claude Code 不同版本里改过几回（完整路径怎么写、支不支持 URL，这些细节都动过）。你正式落地之前，最好先在自己机器的当前版本里实测一下，确认引用能正确加载。万一你那版还不支持，把规则直接分节写在同一个 CLAUDE.md 里也完全没问题，逻辑一点不受影响。

## 七、个人偏好放置区

团队规范管的是"大家一致"。可每个人手头总有些东西，不该硬塞给别人。这些东西有两个去处，别搞混。

CLAUDE.local.md：你对当前这个项目的私人覆盖。它跟团队的 CLAUDE.md 一起躺在项目根目录，但 Claude Code 会自动把它加进 .gitignore，永远不会被提交。这种东西适合放它：

```markdown
# 我的个人配置（只限本项目）

- 我主要负责支付模块（src/payments/）
- 要动 src/payments/ 以外的代码，先跟我确认
- 给我建议命令时带上 -v 详细参数，我想看清细节
- 我本地数据库开在 54321 端口
```

~/.claude/CLAUDE.md：你的跨项目身份。那些走到哪都跟着你的偏好，放这里：

```markdown
# 我的全局偏好（所有项目）

- 用中文回答我
- 改代码前先简单说下方案，等我点头再动手
- 解释技术决策时讲清楚"为什么"，别光说"怎么做"
```

它在所有项目级文件之前加载，一旦冲突，被项目规则盖掉。换句话说，你的个人默认值全局生效，但具体某个项目的约定永远优先。

一句话分清这俩：CLAUDE.local.md 是"我在这个项目里的偏好"，~/.claude/CLAUDE.md 是"我在所有项目里的偏好"。一个跟项目走，一个跟人走。

## 八、用 /memory 检查 Claude 行为

团队协同里报得最多的，就是："新来那同事，他的 Claude 表现得跟我们都不一样。"

查这个，工具就一个。在 Claude Code 里敲：

```text
/memory
```

它会把当前会话实际加载了哪些 CLAUDE.md 文件，全列出来。让出问题那人，和一个一切正常的老成员，各跑一次，两份清单作对比。一般就三种情况：

情况一，新人压根少加载了项目 CLAUDE.md。他清单里干脆没有项目级文件。八成是他从项目根目录以外的某个子目录启动了 Claude Code，或者文件路径不对。让他回根目录重启。

情况二，新人的个人级配置在跟项目规则打架。两份清单都加载了项目文件，可他的 ~/.claude/CLAUDE.md 里，藏着一条跟项目规范冲突的个人指令。个人层有点差异本来很正常，但一旦撞上项目规则，行为就开始不一致。让他翻翻自己的全局配置。

情况三，子目录规则没生效。子目录里的 CLAUDE.md，它不是会话一启动就加载的，而是按需加载，只有当 Claude 真的去读了那个子目录里的某个文件，对应规则才会生效。所以它一开始不会出现在 /memory 列表里。让 Claude 先读一个那个目录下的文件，再跑一次 /memory，你就看到它被加载进来了。这个机制对 monorepo 特别要紧：每个 package 可以有自己的子 CLAUDE.md，但它们是"你进了那块地盘才生效"。

养成个习惯吧：任何"Claude 行为诡异"的报障，第一步都先 /memory。它几乎总能告诉你，Claude 眼里那份规则集，到底长什么样。

## 九、让配置文件慢慢生长

最后一点，也是最容易被忘掉的：一份从来不更新的 CLAUDE.md，会慢慢烂成一张废纸。

代码在变，架构在变，团队踩的坑也在变。一年前写下的某条规则，今天可能早就不适用了，可它还赖在文件里，继续把 Claude 往沟里带。

健康的做法，是让 CLAUDE.md 跟着事故和决策一起长。回头看第四节那个 PR——"金额必须用整数分"那条，不是哪天有人拍脑袋想出来的。它是印尼站结算冒出 0.01 误差、一路查到浮点精度、最后才被沉淀进去的。

CLAUDE.md 就该这么演进：

- 团队踩了个会反复出现的坑 → 走 PR，把它固化成一条"关键陷阱"
- 确立了一条新约定 → 走 PR 补进去
- 发现某条规则过时了 → 走 PR 删掉它

再把"回顾 CLAUDE.md"塞进固定节奏：每月一次也行，每个大版本跟着捋一遍也行。从 Claude Code 里持续榨出价值的团队，我观察下来，从来都不是"配一次就撒手不管"的那种，而是把这份配置当活文档、不停打磨的那种。

## 写在最后

绕回开头我说的那句话。

我做工程师的时候，对"文档"这种东西，老实讲是有点不屑的，觉得代码写干净不就完了，要文档干嘛。转去带出海团队之后，才慢慢感受出来：一个团队真正的瓶颈，常常不在谁的能力，而在那些只长在资深成员脑子里、靠口口相传的隐性知识，压根没被写下来过。新人靠悟，老人靠记，同一个坑换个人再踩一遍。

CLAUDE.md 真正的意义，我觉得不在"配置一个 AI 工具"这么浅显。它做的事情是：把那些隐性的、口头的、飘在空气里的团队约定，变成能版本管理、能 Review、能被一代代继承下去的显性资产。

一个新人 clone 下你们的仓库，他拿到的就不只是代码了。他拿到的是一个已经配置妥当、忠实反映团队真实做法的 Claude。他上班第一天，就站在了团队全部经验的肩膀上。

所以这真不只是"多了一个文件"的事。

一个团队沉淀知识的方式，就是这样开始变化的。
